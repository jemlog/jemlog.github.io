프로젝트를 진행하며 캐시를 사용해 조회 성능을 개선한 경험을 공유하고자 합니다. 

## 조회 성능 개선이 필요한 부분

현재 개선 중인 프로젝트는 사용자가 장소를 방문한 후 같은 그룹의 사람들과 기록을 공유하는 서비스입니다. 기록을 상세 조회 하는 페이지에는 여러 도메인의 정보가 결합되어 전시됩니다.

![스크린샷 2023-12-11 오전 1 20 21](https://github.com/CMC11th-Melly/Melly_Server/assets/82302520/2cf94cb3-7ca6-413b-8cc4-89c629298241)

현재 DDD(도메인 주도 설계)의 개념을 적용해서 도메인 간의 관계를 ID 참조로 구현했습니다. 따라서 연관 데이터를 결합하기 위해서는 도메인 별로 쿼리를 진행해서 최종 정보를 결합해야 합니다. 한번의 요청에 수반되는 쿼리가 많은 만큼
사용자가 증가했을때 얼마나 성능 저하가 발생하는지 테스트가 필요했습니다.

## 성능 테스트 기준 설정

성능 테스트를 진행할때는 `어떤 기준으로 뭘 테스트 해야 하는지` 명확히 정해야 한다고 생각합니다. 

### 응답시간
사용자들이 서비스에서 성능적으로 가장 민감한 것은 무엇일까요? 저는 `응답시간`이라고 생각합니다. 저희는 사용자들에게 최대한 빠른 응답을 제공해줘야 합니다. 따라서 성능테스트가 목표로 삼아야 하는 지표는 응답시간, 그중에서도 상위 90~95%의 응답시간을 일정 수준 이하로 유지하는 것이라 생각합니다.

현재 낮은 서버 스펙을 사용할 수 밖에 없다는 한계(가난합니다...)를 고려하여 응답시간의 임계치는 상위 95% 기준 300ms 정도로 여유 있게 설정하겠습니다. 이때 300ms는 서비스의 `최대 트래픽`을 기준으로 유지되야 합니다.

만약 현재 서비스가 활발히 운영되고 있다면 APM을 분석해서 쉽게 최대 트래픽을 찾을 수 있습니다. 하지만 아직 데이터가 없는 상태에서 부하 테스트를 진행해야 하기에 간단한 계산식을 통해 최대 트래픽을 추정해보겠습니다.

### 1일 총 요청 수
- 계산식 : DAU(Daily Access User) x 1명당 1일 평균 요청 수
- 세팅 : 10만명 x 5번 = 50만

### 1일 평균 RPS
- 계산식 : 1일 총 요청 수 / 하루를 초로 환산
- 세팅 : 50만 / 43200(초) = 11RPS

### 최대 트래픽
- 계산식 : 1일 평균 RPS x 피크 시간 집중률
- 세팅 : 11RPS x 5 = 55RPS

### vUser
- 55명

### DB 세팅
- 유저 : 3만
- 그룹 : 1만
- 메모리 : 30만
- 메모리 이미지 : 60만
- 키워드 : 60만

## 캐시를 적용 적합성 판단

캐시를 적용하기 앞서서 현재 화면이 캐시를 적용하기 적합한지를 판단해보겠습니다.
- 한번 조회하기 위해서 4~5번의 쿼리가 발생한다
- 사용자들이 모두 접근할 수 있는 공개 페이지이므로 조회 빈도가 높다
- 데이터가 자주 변경되지 않는다

위의 특성들을 고려했을때 해당 API에 캐시를 적용하는 것이 합리적이라는 판단을 내렸습니다.
